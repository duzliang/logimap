## 1. 项目目标

### 1.1 总体目标

在未来 6 个月内，将 LogiMap 从「v0.1.0 基础可视化工具」演进为「**v0.3.0 AI 时代的业务逻辑治理平台**」，实现：

1. **团队协作闭环**：团队、权限、审批工作流可用。
2. **逻辑资产沉淀**：版本历史、全局搜索、代码关联可落地。
3. **AI 深度集成**：从「辅助填写」升级为「逻辑治理助手」。
4. **开放与连接**：提供 API / MCP，让 LogiMap 成为研发工具链的一环。

### 1.2 阶段目标

| 阶段 | 版本 | 目标 | 时间周期 |
|------|------|------|---------|
| 第一阶段 | v0.2.0 | 补齐团队协作与治理基础能力 | 6-8 周 |
| 第二阶段 | v0.3.0 | 强化 AI、搜索、代码关联与开放集成 | 8-10 周 |
| 第三阶段 | v0.4.0 | 规模化、企业级与生态扩展 | 10-12 周 |

---

## 2. 开发原则

1. **UI 先行，SPEC 对齐**：所有功能开发前，确认 UI-SPEC.md 和 SPEC.md 中已有对应设计。
2. **数据层先行**：Prisma Schema 变更优先完成，前后端再并行开发。
3. **小步快跑，持续可演示**：每个任务完成后都可在本地运行验证。
4. **AI 作为增强，人不离场**：AI 功能必须有明确的人工确认环节。
5. **测试覆盖关键路径**：认证、权限、节点 CRUD、版本回滚必须有自动化测试。

## 6. 技术债务与质量保障

### 6.1 当前技术债务（基于 v0.1.0 代码审查）

| 债务项 | 描述 | 处理计划 |
|--------|------|---------|
| Dashboard/SystemsPage 样式未统一 | 仍使用旧版 gray/blue 风格 | T2-19 ~ T2-22 统一整改 |
| 部分 UI 组件未按 UI-SPEC | dialog、button、input 等需更新 | T2-19 ~ T2-20 |
| 类型断言较多 | `as any` 在 service 层频繁出现 | 每个版本逐步清理 |
| 缺少单元测试 | 当前测试目录为空 | T2 开始补充 vitest |
| AI 输出解析脆弱 | 依赖正则提取 JSON | T3-1 引入结构化输出/JSON mode |
| 前端路由与布局分散 | 各页面自行实现 header | T2-21 统一 AppShell |

### 6.2 测试策略

| 类型 | 工具 | 覆盖范围 | 目标 |
|------|------|---------|------|
| 单元测试 | vitest | services、validators、lib | 核心逻辑覆盖率 > 80% |
| API 测试 | vitest + supertest | routes | 关键接口覆盖率 > 70% |
| E2E 测试 | Playwright | 登录 → 创建系统 → 创建节点 → 图谱展示 | 主流程通过 |
| 视觉回归 | Storybook + Chromatic（可选） | 核心组件 | 防止 UI 回归 |

### 6.3 代码规范

- 使用 Conventional Commits。
- PR 必须经过至少 1 人 review。
- AI 相关代码必须包含提示词版本和输入输出示例。
- 数据库迁移文件必须附带回滚说明。

## 7. 风险管理

| 风险 | 可能性 | 影响 | 应对策略 |
|------|--------|------|---------|
| AI 功能成本过高 | 中 | 高 | 引入缓存、按团队配额、支持关闭 AI |
| 图谱性能在节点增多后下降 | 高 | 高 | T4-1 性能优化提前到 v0.3.0 做预研 |
| 用户对维护节点缺乏动力 | 高 | 高 | 强化 AI 导入、代码反向生成、CI 集成 |
| 竞品快速跟进 | 中 | 中 | 聚焦细分场景，加速 Agent/代码关联能力 |
| 权限模型设计过复杂 | 中 | 中 | 先实现基础 RBAC，再按需扩展 |
| 第三方 API 变更（Anthropic） | 低 | 中 | 抽象 AI Provider 接口，便于切换模型 |

## 10. 附录

### 10.1 优先级定义

| 优先级 | 含义 |
|--------|------|
| P0 | 阻塞发布，必须完成 |
| P1 | 高价值，尽量完成 |
| P2 | 有价值，资源允许时完成 |

### 10.2 任务优先级汇总

**P0（v0.2.0 必须）**：T2-1 ~ T2-6, T2-7 ~ T2-10, T2-11 ~ T2-14, T2-19 ~ T2-22

### 10.3 迭代节奏建议

- **双周 Sprint**：每个 Sprint 交付可演示的功能。
- **Sprint 计划会**：周一，确定本周任务和验收标准。
- **每日站会**：15 分钟同步进度与阻塞。
- **Sprint 评审**：周五演示，收集反馈。
- **Sprint 回顾**：每两周一次，持续改进流程。
