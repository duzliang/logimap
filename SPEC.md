## 1. 产品概述

### 1.1 一句话定义

LogiMap 是面向研发团队的**业务逻辑可视化与治理平台**，通过结构化节点与交互式图谱，把散落在代码、文档、会议、人脑中的业务知识沉淀为可检索、可评审、可追溯的逻辑资产。

### 1.2 产品愿景

在 AI 编程 Agent 大规模进入研发流程的 2026 年，代码生成速度已不再是瓶颈，真正的瓶颈变成了：

- **逻辑理解**：Agent 和人类如何快速理解「这段代码为什么这么写」；
- **变更治理**：一次改动会触发哪些业务规则、依赖、边界条件；
- **知识沉淀**：当人员流动、模型迭代时，组织能否保留关键业务决策。

LogiMap 的愿景是成为**研发团队的「业务逻辑层」**——不替代代码，也不替代文档，而是作为代码与业务意图之间的权威映射层，让人和 AI 都能快速理解、验证、追溯业务逻辑。

### 1.3 核心价值主张

| 价值 | 说明 |
|------|------|
| **可视化理解** | 用 React Flow 交互图谱把复杂业务逻辑变成可浏览、可搜索的知识网络 |
| **AI 辅助但不替代人** | Claude 辅助生成内容、发现边界、生成测试用例，最终由人评审确认 |
| **状态可追踪** | 节点从草稿 → 评审 → 确认 → 废弃的全生命周期管理 |
| **代码可追溯** | 每个逻辑节点都可关联到具体代码位置，支持双向跳转 |
| **变更可预测** | 改动节点时自动提示关联节点、下游依赖和潜在影响 |

## 2. 市场背景与需求迭代

### 2.1 市场环境（2026）

- **业务规则管理市场（BRMS）**：2026 年约 **24.8 亿美元**，预计 2031 年达到 36.6 亿美元（Mordor Intelligence）。
- **通用图表/可视化工具市场**：2026 年约 **12.6 亿美元**，预计 2030 年达到 18.8 亿美元（The Business Research Company）。
- **AI 编程 Agent 普及**：Claude Code、Cursor、GitHub Copilot Workspace 等工具已从「代码补全」演进为可执行多文件改动的 Agent。

### 2.2 真实场景中的痛点

1. **Agent 生成代码后，业务逻辑没人整理**
   - Agent 能快速产出代码，但业务规则、分支条件、边界处理往往散落在多次对话和提交中。
   - 结果：代码越写越快，但「为什么这样写」越来越不透明。

2. **代码改动的影响范围难以判断**
   - 微服务、事件驱动、复杂状态机下，一个字段变更可能触发多条业务线。
   - 现有工具（Jira、Confluence、白板）无法提供**结构化的影响分析**。

3. **评审成为新瓶颈**
   - AI 生成代码量增加后，人类评审负担加重。
   - 需要一种「逻辑先看懂，再看代码」的评审模式。

4. **组织知识随人员流动而流失**
   - 关键业务规则存在于老员工的脑中或历史 PR 中，新人和 Agent 都难以获取。

### 2.3 LogiMap 的机会定位

| 竞品/替代方案 | 局限 | LogiMap 差异 |
|--------------|------|-------------|
| 通用白板（Miro、Whimsical、Eraser.io） | 自由度高但缺乏结构化，难以沉淀为可执行知识 | 以「逻辑节点」为最小单位，强制结构化字段 |
| 文档工具（Confluence、Notion、Lark） | 线性文档，难以表达复杂依赖关系 | 图谱 + 文档双视图，节点即知识单元 |
| BRMS（FICO、IBM ODM） | 重、贵、偏向业务人员，研发接入成本高 | 轻量、研发原生、与代码仓库自然关联 |
| AI Coding Agent（Claude Code、Cursor） | 强在生成代码，弱在沉淀和组织逻辑 | 与 Agent 协作，输出可被 Agent 读取的上下文 |

### 2.4 需求迭代方向

基于以上市场分析，LogiMap 从「可视化工具」升级为「**AI 时代的业务逻辑治理基础设施**」，核心迭代方向：

1. **逻辑即代码资产**：节点可与 Git 仓库、代码文件、函数签名深度关联。
2. **Agent 可读上下文**：支持导出 `.cursorrules`、`AGENTS.md`、`SKILLS.md` 等 Agent 指令文件。
3. **影响分析与变更治理**：改动节点时自动计算影响范围（blast-radius）。
4. **人机协作评审**：强化草稿 → 评审 → 确认工作流，降低 AI 生成代码的评审成本。
5. **开放集成**：提供 API / MCP（Model Context Protocol），让外部 Agent 和工具能读取/写入逻辑图谱。

### 3.1 主要用户角色

| 角色 | 职责 | 使用场景 |
|------|------|---------|

**李姐，技术负责人**
- 收到需求：修改优惠券计算规则。

- 发现改动会影响「订单结算」「退款金额计算」「会员价叠加」三个节点。
- 在图谱上标注影响范围，安排相关工程师评估。

## 4. 核心概念

### 4.1 实体关系

```
Team ──▶ System ──▶ Module ──▶ LogicNode ──▶ Connection
  │         │
  │         └── 业务系统（如「订单系统」）
  └── 团队（成员、权限）

LogicNode：业务逻辑的最小单元
Connection：节点之间的关系（触发 / 依赖 / 阻断 / 扩展）
LogicNodeVersion：节点历史快照
```

### 4.2 逻辑节点（LogicNode）

一个逻辑节点承载一个完整的业务逻辑单元，包含 6 个核心字段：

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `trigger` | 富文本 | 触发条件 | 维修工单状态变为「已完成」 |
| `dependsOn` | 富文本 | 前置依赖 | 客户已确认结算单 |
| `mainFlow` | 富文本 | 主流程步骤 | 1. 计算工时费 2. 计算材料费 3. 生成结算单 |
| `branches` | JSON 数组 | 分支条件 | [{condition:"会员", action:"应用会员价"}] |
| `edgeCases` | JSON 数组 | 边界处理 | [{scenario:"金额为 0", handling:"跳过支付", severity:"warning"}] |
| `codeRef` | 字符串 | 代码关联 | `src/services/settlement.ts#calculateSettlement` |

### 4.3 节点状态

| 状态 | 含义 | 颜色（见 UI-SPEC） |
|------|------|------------------|
| **DRAFT** | 草稿，待完善 | 灰色 |
| **REVIEW** | 已提交评审 | 琥珀色 |
| **APPROVED** | 已确认，可作为权威逻辑 | 翠绿色 |
| **DEPRECATED** | 已废弃，保留历史 | 玫瑰色 |

### 4.4 节点关系（Connection）

| 类型 | 含义 | 视觉样式（见 UI-SPEC） |
|------|------|----------------------|
| **TRIGGERS** | A 触发 B | 紫色实线 |
| **DEPENDS_ON** | A 依赖 B | 灰色虚线 |
| **BLOCKS** | A 阻断 B | 红色实线 |
| **EXTENDS** | A 扩展 B | 绿色点划线 |

## 5. 功能需求

### 5.1 用户与团队（v0.2.0）

- **FR-A1**：用户可通过邮箱注册、登录、登出，使用 JWT 认证。
- **FR-A2**：系统支持多团队（Team），每个团队拥有独立的系统空间。
- **FR-A3**：团队成员角色包括：OWNER、ADMIN、MEMBER、VIEWER。
- **FR-A4**：支持邀请成员、移除成员、转让团队所有权。

### 5.2 系统与模块管理（v0.1.0 已完成，v0.2.0 增强）

- **FR-B1**：可在团队下创建业务系统，设置名称、标识、描述、图标、颜色。
- **FR-B2**：可在系统下创建模块，设置名称、标识、描述、排序、颜色。
- **FR-B3**：模块列表支持拖拽排序。
- **FR-B4**：系统/模块支持权限控制（OWNER/ADMIN 可编辑，MEMBER 可编辑节点，VIEWER 只读）。

### 5.3 逻辑节点管理（v0.1.0 已完成，v0.2.0 增强）

- **FR-C1**：可在模块下创建、编辑、删除逻辑节点。
- **FR-C2**：节点包含 6 个核心字段：触发条件、前置依赖、主流程、分支条件、边界处理、代码关联。
- **FR-C3**：节点支持标签、备注、优先级、状态字段。
- **FR-C4**：主流程支持富文本编辑（TipTap），可插入有序/无序列表、加粗、链接等。
- **FR-C5**：节点编辑支持草稿自动保存，防止误刷新丢失内容。

### 5.4 逻辑图谱（v0.1.0 已完成，v0.2.0 增强）

- **FR-D1**：以 React Flow 交互式图谱展示模块内的逻辑节点和关系。
- **FR-D2**：支持拖拽节点、缩放画布、平移、MiniMap、Controls。
- **FR-D3**：支持手动连线创建节点关系。
- **FR-D4**：支持自动布局（Dagre）。
- **FR-D5**：支持导出图谱为 PNG（已具备），后续支持 SVG / PDF。
- **FR-D6**：双击节点打开详情，右键/工具栏进入编辑。
- **FR-D7**：节点状态颜色在图谱中实时反映（见 UI-SPEC 节点状态色）。

### 5.5 AI 辅助功能（v0.1.0 已完成，v0.2.0 增强）

- **FR-E1**：根据节点名称和模块上下文，AI 生成触发条件、前置依赖、主流程、分支、边界。
- **FR-E2**：AI 分析节点完整性，给出完整性评分和改进建议。
- **FR-E3**：AI 基于主流程建议遗漏的边界条件。
- **FR-E4**：AI 基于节点内容生成测试用例（正常、边界、分支）。
- **FR-E5**（新增）：AI 根据节点关联的代码引用，反向检查代码与逻辑描述是否一致。
- **FR-E6**（新增）：支持将逻辑图谱导出为 `.cursorrules` / `AGENTS.md` 片段，供 AI Agent 读取。

### 5.6 版本与审批工作流（v0.2.0）

- **FR-F1**：节点每次保存自动生成版本快照。
- **FR-F2**：支持查看版本历史、对比差异、回滚到指定版本。
- **FR-F3**：节点状态转换：DRAFT → REVIEW → APPROVED / DEPRECATED。
- **FR-F4**：状态转换需由有权限的成员执行，并记录操作人和备注。
- **FR-F5**：节点进入 REVIEW 时，自动通知相关成员。

### 5.7 搜索与发现（v0.2.0）

- **FR-G1**：全局搜索节点、模块、系统。
- **FR-G2**：支持按标签、状态、优先级、负责人筛选节点。
- **FR-G3**：图谱支持高亮匹配节点和关联路径。

### 5.8 影响分析（v0.3.0）

- **FR-H1**：当编辑或删除节点时，自动计算并展示受影响的上下游节点。
- **FR-H2**：支持「假设分析」：在不保存的情况下模拟改动的影响范围。
- **FR-H3**：影响分析结果可导出为变更影响报告。

### 5.9 代码关联与集成（v0.3.0）

- **FR-I1**：节点 `codeRef` 支持文件路径、函数名、GitHub/GitLab 永久链接。
- **FR-I2**：支持从 Git 仓库导入目录结构，辅助创建系统/模块。
- **FR-I3**：支持生成 OpenAPI / Swagger 文档。
- **FR-I4**：提供 REST API 和 MCP 接口，供外部 Agent 查询逻辑图谱。

### 5.10 通知与协作（v0.3.0）

- **FR-J1**：节点状态变更、被提及、评审请求时发送站内通知。
- **FR-J2**：支持邮件通知（可选）。
- **FR-J3**：支持多人实时协作编辑图谱（WebSocket，远期）。

## 6. 数据模型

数据模型已在 Prisma Schema 中定义，核心实体如下：

```prisma
User ──▶ TeamMember ◀── Team
                        │
                        └──▶ System ──▶ Module ──▶ LogicNode
                                                     │
                                                     ├──▶ Connection
                                                     └──▶ LogicNodeVersion
```

### 6.1 关键字段说明

- **User**：`id`, `email`, `name`, `passwordHash`, `avatarUrl`
- **Team**：`id`, `name`, `slug`, `description`
- **TeamMember**：`role` ∈ {OWNER, ADMIN, MEMBER, VIEWER}
- **System**：`id`, `name`, `slug`, `description`, `icon`, `color`, `teamId`
- **Module**：`id`, `name`, `slug`, `description`, `order`, `color`, `systemId`
- **LogicNode**：`id`, `name`, `summary`, `status`, `priority`, `trigger`, `dependsOn`, `mainFlow`, `branches`, `edgeCases`, `codeRef`, `positionX`, `positionY`, `tags`, `notes`, `moduleId`, `createdById`, `updatedById`
- **Connection**：`id`, `label`, `type`, `description`, `sourceId`, `targetId`
- **LogicNodeVersion**：`id`, `version`, `snapshot`, `changeNote`, `nodeId`

### 6.2 版本快照策略

- 每次 `update` 节点时，在事务中先创建 `LogicNodeVersion` 快照，再更新节点。
- 回滚时，先保存当前状态为新版本，再恢复快照内容。
- 保留最近 50 个版本，更早版本可归档或清理。

## 7. AI 能力设计

### 7.1 当前 AI 能力（v0.1.0）

| 能力 | 输入 | 输出 | 模型 |
|------|------|------|------|
| 生成节点内容 | 节点名称、模块上下文、已有内容 | 触发、依赖、主流程、分支、边界 | Claude Sonnet 4 |
| 分析完整性 | 节点全部字段 | 完整性评分、改进建议、遗漏边界、建议分支 | Claude Sonnet 4 |
| 建议边界条件 | 节点名称、主流程、已有边界 | 最多 5 个边界条件建议 | Claude Sonnet 4 |
| 生成测试用例 | 节点全部字段 | 正常/边界/分支测试用例 | Claude Sonnet 4 |

### 7.2 规划 AI 能力（v0.2.0+）

| 能力 | 场景 | 说明 |
|------|------|------|
| **代码-逻辑一致性检查** | 当 `codeRef` 存在时，读取代码并与节点描述对比，提示不一致 | 降低 AI 生成代码与文档脱节的风险 |
| **影响分析辅助** | 改动节点时，AI 建议可能受影响的下游节点 | 结合图谱结构 + 语义相似度 |
| **Agent 上下文导出** | 导出 `.cursorrules` / `AGENTS.md` | 让 Agent 在编码时参考业务逻辑 |
| **自然语言查询** | 用户用中文/英文提问，AI 在图谱中检索并回答 | 例如：「退款有哪些边界条件？」 |
| **批量生成节点** | 根据 PRD/需求文档，批量建议模块和节点结构 | 用于新项目初始化 |

### 7.3 AI 使用原则

- **人类最终确认**：AI 生成内容默认进入草稿状态，需人工评审后才能确认。
- **可追溯**：AI 生成的内容需记录生成时间、模型版本、输入摘要。
- **可关闭**：团队管理员可关闭 AI 功能，或限制仅特定角色可用。
- **安全**：AI 不直接访问生产数据库或执行代码，仅读取/生成文本内容。

## 8. 非功能需求

### 8.1 性能

- **NFR-1**：图谱首次加载时间 < 2s（100 个节点以内）。
- **NFR-2**：节点保存响应时间 < 500ms。
- **NFR-3**：AI 生成响应时间 < 10s（含网络往返）。
- **NFR-4**：支持图谱虚拟滚动或懒加载，以应对 500+ 节点场景。

### 8.2 可用性

- **NFR-5**：界面语言优先中文，后续支持英文（i18n）。
- **NFR-6**：核心操作流程（创建节点 → 编辑 → 保存 → 提交评审）可在 3 步内完成。
- **NFR-7**：所有交互元素满足 WCAG AA 对比度要求（见 UI-SPEC 第七层）。

### 8.3 安全

- **NFR-8**：用户密码使用 bcryptjs 哈希存储。
- **NFR-9**：JWT 使用 jose 库签名，支持过期与刷新（后续版本）。
- **NFR-10**：API 权限校验到团队/系统/模块级别，防止越权访问。
- **NFR-11**：AI 调用不传递用户敏感数据，API Key 仅服务端持有。

### 8.4 可维护性

- **NFR-12**：前后端使用 TypeScript，核心类型共享。
- **NFR-13**：API 遵循 RESTful 风格，后续提供 OpenAPI 文档。
- **NFR-14**：数据库迁移使用 Prisma Migrate，版本可回滚。

### 8.5 部署

- **NFR-15**：支持 Docker Compose 一键部署。
- **NFR-16**：前端为静态资源，可由 Nginx 承载并反向代理 API。
- **NFR-17**：后端为无状态服务，可水平扩展。

## 9. 用户故事与验收标准

### US-1：创建业务系统

**作为** 团队管理员，**我希望** 创建一个新的业务系统，**从而** 组织团队内的逻辑资产。

**验收标准**：
- 可填写系统名称、标识、描述、图标、颜色。
- 标识只允许小写字母、数字和连字符。
- 同一团队下标识唯一。
- 创建成功后自动进入系统详情页。

### US-2：编写逻辑节点

**作为** 后端工程师，**我希望** 用结构化字段编写业务逻辑，**从而** 把隐性的业务规则显性化。

**验收标准**：
- 必填字段：节点名称。
- 可选字段：概述、触发条件、前置依赖、主流程、分支条件、边界处理、代码关联、标签、备注。
- 分支和边界可动态增删。
- 保存后节点进入 DRAFT 状态。

### US-3：AI 辅助生成内容

**作为** 后端工程师，**我希望** AI 根据节点名称生成业务逻辑草稿，**从而** 减少从零开始的成本。

**验收标准**：
- 点击「AI 生成」后，AI 返回触发、依赖、主流程、分支、边界建议。
- 生成的内容填充到对应字段，用户可编辑。
- AI 未返回有效 JSON 时，提示错误信息。

### US-4：可视化逻辑图谱

**作为** 技术负责人，**我希望** 在图谱上查看模块内所有节点及关系，**从而** 理解业务逻辑的全貌。

**验收标准**：
- 节点按状态显示不同边框颜色。
- 支持拖拽、缩放、平移。
- 支持自动布局。
- 双击节点打开详情弹窗。

### US-5：提交节点评审

**作为** 后端工程师，**我希望** 把节点从草稿提交为评审状态，**从而** 让技术负责人确认。

**验收标准**：
- DRAFT 节点可转为 REVIEW。
- REVIEW 节点可被 APPROVED 或退回 DRAFT。
- 状态转换记录操作人和时间。
- 状态变更后通知相关成员。

### US-6：代码关联跳转

**作为** 开发者，**我希望** 节点能关联到代码文件，**从而** 在逻辑和实现之间快速切换。

**验收标准**：
- `codeRef` 支持相对路径、函数名、GitHub/GitLab 永久链接。
- 在节点详情中点击 codeRef 可打开对应链接。
- 后续支持从代码仓库反向查看关联的节点。

### US-7：变更影响分析

**作为** 技术负责人，**我希望** 修改节点前看到影响范围，**从而** 避免遗漏下游依赖。

**验收标准**：
- 编辑节点时，提示直接连接的上下游节点。
- 支持递归展示间接依赖（最多 3 层）。
- 影响范围可导出为报告。

## 11. 风险与假设

### 11.1 关键假设

- 用户愿意花时间维护业务逻辑节点，而非完全依赖自动提取。
- 研发团队有「逻辑可视化」的痛点，且现有工具无法很好解决。
- AI 生成内容的质量足够高，能显著减少人工编写时间。

### 11.2 主要风险

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 用户不愿维护节点，导致数据稀疏 | 产品价值下降 | AI 批量导入、代码反向生成、与 CI/CD 集成 |
| AI 生成内容质量不稳定 | 用户信任度下降 | 支持人工编辑、评分反馈、模型版本管理 |
| 图谱复杂后性能下降 | 体验变差 | 虚拟滚动、分层展示、子图折叠 |
| 竞品（Linear、Notion、Cursor）推出类似功能 | 差异化被削弱 | 专注「业务逻辑治理」细分领域，强化代码关联与 Agent 集成 |
